## Goal

Split by **domain responsibilities**, not by “file count”.

### Good split (simple + scalable)

1. **FormTemplatesService**

* create template (draft)
* list templates (admin)
* get template by id
* publish / archive
* clone (versioning + familyId)
* update draft (replace fields)

2. **FormSubmissionsService**

* create submission (answer validation)
* list submissions (filters/pagination)
* get submission by id (role-based access)
* list my submissions

3. **SubmissionReviewService**

* approve submission
* reject submission

4. **FormsPolicy / AccessService** (tiny helper)

* “can this user access plantId?”
* “admin vs manager vs user rules”
* keeps controllers/services readable

5. **AnswerValidationService** (tiny helper)

* `validateAnswerValue(field, value)`
* later: dropdown options, photo key format, etc.

That’s it. This is already “grown-up” architecture.

## Folder structure suggestion

`services/api/src/forms/`

* `forms.module.ts`
* `controllers/`

  * `forms.controller.ts` (admin)
  * `mobile-forms.controller.ts` (mobile)
* `services/`

  * `form-templates.service.ts`
  * `form-submissions.service.ts`
  * `submission-review.service.ts`
  * `forms-policy.service.ts` (or `forms-access.service.ts`)
  * `answer-validation.service.ts`
* `dto/` (keep as is)
* `types/` (optional)

Entities stay in `src/entities/forms/*` as you have.

## What stays in controllers vs services

Controllers should be dumb:

* read params/body
* get `me` from `@CurrentUser`
* call the right service
* return result

All rules live in services/policy.

---

## How to do it without breaking anything (safe migration steps)

1. **Create new services** and *move code* (cut/paste) out of `FormsService` gradually.
2. Keep `FormsService` temporarily as a “facade” that delegates:

   * `this.templatesService.create(...)`
   * `this.submissionsService.list(...)`
3. Once everything moved, delete the old `FormsService`.

This avoids a big-bang refactor.
